I want to design a scalable and resilient event-processing system that can handle ~5,000 messages per second per upstream without Kafka rebalancing/partition/concurrency issues. The system must support retries of failed messages, a strong and consistent DB schema, and all other factors needed for scaling.

Context:
- 5–6 unique upstream input channels, each posting messages onto Kafka topics.
- Business logic for validation, normalization, and processing varies per upstream.
- Processed messages are dispatched to SMS, Email, OTP, Voice, Webhooks, and Push Notifications (via enterprise vendor APIs).
- Target stack: Java 17, Gradle, Spring Boot 3.x, Spring Kafka, Oracle DB 19c, Redis (only if absolutely required), Resilience4J, Logback.
- Deployment: OpenShift (OCP), with configurations in ConfigMap and `application.yml`.

Requirements:
- Workflow must be configuration-driven and pluggable (adding new upstream should only require business logic + config changes).
- Incorporate industry-standard patterns: orchestration, retry/DLQ, factory/adaptor, throttling & rate-limiting, async processing, audit, caching, transactional guarantees, etc.
- Follow Kafka best practices for large-scale processing.
- Proper design of DB tables with normalization and considerations in using DB sequence (e.g., should we use NUMBER(19)?)
- Clear error codes defined and used in every stage/step of message processing (proper audit trail)
